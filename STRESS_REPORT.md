# ThriftAI Stress Report

Where ThriftAI helps, where it hurts, and how to tune it.

## TL;DR

- **Frontier models (Sonnet, GPT-4o):** caching is profitable above ~0.3% hit rate. Almost always wins.
- **Mid-tier models (Haiku, GPT-4o-mini):** caching profits above ~3-5% hit rate. Profitable in most realistic agent workloads.
- **Tiny queries (< 100 chars) on cheap models:** the embedding round-trip dominates. Use the default `min_query_chars=100` skip; below the threshold the semantic cache is a no-op.
- **High-cardinality inputs (every request unique):** cache hit rate ≈ 0; you pay overhead for nothing. Set `Session(enabled=False)` or `THRIFTAI_DISABLED=1`.
- **Wrong-hit risk:** the semantic cache reliably handles paraphrases but **does not** distinguish negation, numeric drift, or entity swaps when surrounding context is shared. See `tests/adversarial_report.md` for per-category numbers at the default threshold.

## Cost crossover

Per-query expected savings:

```
E[savings] = p × C_LLM − C_embed
```

where `p` is the empirical cache hit rate. Break-even hit rate:

```
p_breakeven = C_embed / C_LLM
```

Above `p_breakeven`, ThriftAI saves money. Below, it costs money. The constant numbers below assume `text-embedding-3-small` at $0.02 per 1M tokens (the default for ThriftAI's semantic cache when configured with `embedding_model="text-embedding-3-small"`).

| Model | Input tokens | Output tokens | C_LLM | C_embed | **p_breakeven** |
|---|---:|---:|---:|---:|---:|
| `claude-sonnet-4` | 500 | 200 | $0.0045 | $0.00001 | **0.22%** |
| `gpt-4o` | 500 | 200 | $0.0033 | $0.00001 | **0.31%** |
| `claude-haiku-4-5` | 500 | 200 | $0.00038 | $0.00001 | **2.67%** |
| `gpt-4o-mini` | 500 | 200 | $0.00020 | $0.00001 | **5.13%** |
| `claude-haiku-4-5` | 2000 | 500 | $0.0011 | $0.00004 | **3.56%** |
| `claude-haiku-4-5` | 50 | 20 | $0.000038 | $0.000001 | **2.67%** |
| `gpt-4o-mini` | 50 | 20 | $0.0000195 | $0.000001 | **5.13%** |

The numbers above are computed in `tests/test_cost_crossover.py` from synthetic per-token rates that match commercial pricing as of 2026-Q2. The tests run on every PR, so any regression in the formula is caught automatically. Real LiteLLM pricing data may drift slightly; if it does, update `PRICING` in the test file in a single place.

### Reading the table

**A 5% break-even on `gpt-4o-mini` is not a problem in practice** — most realistic agent workloads cache at 30–80% hit rates. The break-even number tells you the *floor* below which ThriftAI loses money. Use it as a sanity check, not a target.

The table also shows that **break-even is roughly constant across query sizes for a given model** — both input and output token costs scale linearly, and so does embedding cost. The `min_query_chars` skip exists for *latency* and *embedding-quality* reasons, not pure arithmetic break-even (see below).

## Knobs (all on `Session`)

| Param | Default | What it does |
|---|---|---|
| `enabled` | `True` | Master kill switch. `False` makes the whole library a no-op (cost tracking still works). Honors `THRIFTAI_DISABLED=1` env var. |
| `embedding_model` | `None` | Set to enable semantic cache. `None` keeps it off (exact cache only). |
| `semantic_threshold` | `0.92` | Cosine similarity threshold for a hit. Higher = fewer wrong-hits, more false misses. See `tests/adversarial_report.md` for per-category trade-offs. |
| `semantic_min_chars` | `100` | Skip the embedding round-trip entirely for queries shorter than this. Saves the round-trip latency on tiny inputs where caching has near-zero value. Set to `0` to disable the skip. |
| `semantic_bucket_size` | `1000` | Cap entries per `(agent, prompt_hash)` bucket. FIFO eviction by `created_at`. Prevents the linear similarity scan from going quadratic in long-running sessions. |

## Why `min_query_chars` exists

Pure-math break-even doesn't strictly require the skip — input and output token costs both scale linearly, so `C_embed / C_LLM` is roughly constant across query sizes. The real justifications:

1. **Latency.** An embedding round-trip adds 50–200 ms even on a hit. For sub-100-char queries, that's often slower than the LLM call itself would be.
2. **Cardinality.** Short queries (function calls, tool names, status codes, date strings) are usually unique per call; cache hit rate ≈ 0, so every embedding is wasted.
3. **Embedding quality.** Short text produces noisy embeddings; similarity scores are unreliable below ~10 tokens.

If your workload has long queries (e.g. RAG with 1k-token chunks), consider raising `semantic_min_chars` to 200+. If you have short, *recurring* queries (rare but possible), lower it.

## Wrong-hit risk

See [`tests/adversarial_report.md`](./tests/adversarial_report.md) — generated by running `tests/test_adversarial.py` against a real embedding model. It shows per-category cosine similarity at thresholds 0.85, 0.92, 0.95.

**Headline: at the default threshold of 0.92,** the semantic cache reliably catches paraphrases but **does not** distinguish:
- Negation (`is X?` vs `is X not?`) — embeddings consistently score these as near-identical.
- Numeric drift (`Compute 2+2` vs `Compute 2+3`) — lexical similarity dominates.
- Entity swaps (`CEO of Apple` vs `CEO of Google`) — single proper-noun changes get diluted.
- Date drift (`Q1 2024` vs `Q1 2025`) — same risk class.

If your workload includes any of those — financial Q&A, scientific computation, real-time entity lookup — **either raise the threshold to 0.95+ or disable semantic caching for the affected agents** (you can scope it per agent: instantiate two `Session`s with different `embedding_model` settings, or use the exact cache only by leaving `embedding_model=None`).

## Operational gotchas

These are tracked but not solved in v0.1:

- **Single-instance SQLite.** Each replica has its own cache file; there's no shared state across servers in a horizontal deployment. Plan: Redis backend in v0.3.
- **No TTL** on cached responses. Invalidate manually with `cache.invalidate_agent(name)` after a model upgrade or data refresh.
- **Response text stored unencrypted** on disk. If your agent inputs contain PII, encrypt the cache directory at rest or run with `enabled=False`.
- **Linear scan within a bucket** — capped via `bucket_size` (default 1000). Above ~10k entries per bucket, the scan starts to bite. The scale tests in `tests/test_scale.py` (gated on `pytest -m slow`) assert p95 lookup <100 ms at 10k entries.
